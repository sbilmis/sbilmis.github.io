1. Difference Between `source script.sh` and `sh script.sh`

  • `source script.sh` or `. script.sh`:
    • Execution Context: The script is executed in the current shell, affecting its environment directly.
    • Environment Variables: Any changes (e.g., setting variables, changing directories) persist in the current shell after the script finishes.
    • Command Type: Built-in shell command.
  • `sh script.sh`:
    • Execution Context: Runs the script in a new shell process (subshell), isolating its effects from the current shell.
    • Environment Variables: Changes made by the script only affect the subshell and are discarded once the script ends.
    • Compatibility: Ensures the script is executed using `sh`, regardless of the current shell.
  • Summary:
    • `source script.sh` executes the script in the current shell, impacting its environment.
    • `sh script.sh` runs the script in a separate subshell, isolating its changes.

2. How to Create a Multiboot USB

  • What is a Multiboot USB?
    • A Multiboot USB allows multiple bootable ISO images (e.g., operating systems, recovery tools) to coexist on a single USB drive. It is useful for:
      • Installing various operating systems.
      • Running live environments (e.g., Linux distributions).
      • Using recovery and diagnostic tools like Clonezilla, GRML, or Knoppix.
  • Tool: YUMI
    • YUMI (Your Universal Multiboot Installer) is a user-friendly tool that works on both Linux and Windows to create Multiboot USB drives.
    • It supports a wide range of ISO images.
  • Steps:
    1. Download and install YUMI from pendrivelinux.com.
    2. Launch YUMI and select your USB drive.
    3. Add ISO images one by one using the interface.
    4. Boot from the USB drive and select the desired ISO from the boot menu.
  • Use Case Examples:
    • Install Linux distributions like Ubuntu, Fedora, or Debian.
    • Run system recovery tools such as Clonezilla or GRML.
    • Carry multiple OS installers for convenience.

4. How to Use Cron Jobs

  • Objective: Automate the execution of tasks or scripts at specific intervals using the cron service.
  • Key Commands:
# Edit the user's crontab file
crontab -e
# List current cron jobs
crontab -l
  • Example Cron Job:
30-45 16 * * * /Users/sbilmis/Ninbox/truba_scripts/cron_test.sh > /Users/sbilmis/Ninbox/truba_scripts/test.txt 2> /Users/sbilmis/Ninbox/truba_scripts/testerr.txt
# This runs the script `cron_test.sh` between 16:30 and 16:45 daily.
# `>` redirects standard output (stdout) to test.txt.
# `2>` redirects errors (stderr) to testerr.txt.
  • Cron Time Syntax:
  • `* * * * *`: (minute, hour, day, month, day of the week)
  • Example shortcuts:
    • `@reboot`: Run at system startup.
    • `@hourly`, `@daily`, `@weekly`, `@monthly`, `@yearly`: Run at specified intervals.
  • Special Characters:
  • `,`: Separate list items (e.g., `0 14,20 * * *` for 14:00 and 20:00).
  • `-`: Define a range (e.g., `30-45` for minutes 30 to 45).
  • Important Notes:
  • On macOS, grant Full Disk Access to cron for it to function properly:
    • System Preferences → Security & Privacy → Full Disk Access.
  • In Linux, ensure the executable's full path is specified if `PATH` is not set in the crontab file.
  • For systems that sleep, use `rtcwake` in Linux or launchd in macOS for more reliability.
  • Define PATH in Crontab:
SHELL=/bin/sh
PATH=/opt/homebrew/bin/:/usr/local/sbin:/usr/local/bin:/sbin:/bin:/usr/sbin:/usr/bin
19-23 12 * * * /path/to/your_script.sh >> /path/to/log.txt 2>> /path/to/error.txt

5. How to Redirect Both Output (stdout) and Errors (stderr) to Files

  • Objective: Capture both the standard output (stdout) and error output (stderr) of a command.
  • Syntax:
# Redirect stdout to a file and stderr to a different file
command1 > out.txt 2> err.txt

# Redirect both stdout and stderr to the same file
command1 > everything.txt 2>&1
  • Explanation:
  • `>` redirects stdout.
  • `2>` redirects stderr.
  • `2>&1` merges stderr into stdout, sending both to the same file.
  • Descriptors:
  • `0`: stdin (standard input)
  • `1`: stdout (standard output)
  • `2`: stderr (standard error)
  • Examples:

  • Save the output of a script while capturing any errors separately: 

    myscript.sh > output.log 2> error.log

  • Combine both outputs into a single log file: 

    myscript.sh > combined.log 2>&1

6. How to Share Folder Between Guest and Host in virt-manager (KVM)

  • Objective: Share a folder between the host and guest operating systems in KVM using virt-manager.
  • Steps to Configure:
    1. Power off the guest machine.
    2. Open the virtual machine console in virt-manager:
      • Navigate to Hardware Details → Memory → Enable Shared Memory.
    3. Click the "Add Hardware" button:
      • Select File System.
      • Set the Driver to `virtio-9p`.
      • Browse to select the host path to share.
      • Specify a target path name (e.g., `tagmyfolder`).
    4. Start the guest machine.

  • Mount the Shared Folder in the Guest Machine: 

    # Create a directory to mount the shared folder
sudo mkdir /mnt/myfolder

# Mount the folder using the virtio driver
sudo mount -t 9p -o trans=virtio tag_my_folder /mnt/myfolder

# Alternative command for newer Ubuntu versions
sudo mount -t virtiofs tag_my_folder /mnt/myfolder
  • Notes:
    • Replace `tagmyfolder` with the target path name you specified in virt-manager.
    • Use `virtiofs` if `virtio-9p` is not supported in your system.
  • References:
    • [Debugpoint: Share Folder in virt-manager](https://www.debugpoint.com/share-folder-virt-manager/)
    • [QEMU 9p Setup Documentation](https://wiki.qemu.org/Documentation/9psetup#Mounting_the_shared_path)
    • [UTM GitHub Issue #4386](https://github.com/utmapp/UTM/issues/4386)

7. How to Use Yum History to Roll Back an Update in Red Hat Enterprise Linux 6 and Later

  • Objective: Roll back or downgrade a package update using yum history in CentOS/RHEL.
  • Steps to Roll Back an Update:

    1. Identify the transaction ID of the update to be undone: 

      # List yum transaction history
yum history

    2. Use the `undo` command with the identified transaction ID: 

      # Roll back the update using transaction ID (e.g., 8)
yum history undo 8
  • Alternative: Downgrade a Package

    • If rolling back is not an option or specific version control is needed: 

      # Downgrade a package to a specific version
yum downgrade httpd-2.2.3-22.el5
  • Notes:
    • The `yum history` command displays all recent transactions with their IDs.
    • Rolling back may also undo dependencies installed during the transaction.
    • Downgrading requires the specific version of the package to be available in the repository.
  • References:
    • [Red Hat Documentation: Roll Back an Update Using Yum History](https://access.redhat.com/solutions/64069link)

8. Yum Priorities

  • Objective: Manage and troubleshoot yum priorities, especially when packages are excluded due to repository priority protections.
  • Steps:

    1. Disable Yum Priorities Temporarily: 

      # Use the following option to disable yum priorities for a single command
yum --disableplugin=priorities

    2. Disable Yum Priorities Globally: 

      # Edit the yum priorities configuration file
sudo nano /etc/yum/pluginconf.d/priorities.conf

# Set the following:
enabled = 0
    3. Identify Excluded Packages:

      • To check which packages are excluded due to priority protections: 

        # List updates with detailed debug information
yum list updates --debuglevel 3
  • Notes:
    • Yum priorities help manage package selection when multiple repositories provide the same package. Higher priority repositories are favored.
    • Disabling priorities can temporarily allow access to lower-priority packages, which may help resolve dependency issues.

9. Yum Repositories

  • Objective: Manage yum repositories for package updates effectively.
  • Steps:
    1. View All Available Repositories:

      • Use the following command to list all enabled repositories on your system: 

        yum repolist
    2. Exclude a Repository Temporarily:

      • If you want to exclude a specific repository (e.g., `mongodb-org-3.0`) during an update: 

        yum --disablerepo=mongodb-org-3.0 update
    3. Exclude Multiple Repositories Temporarily:

      • To exclude multiple repositories in a single update command: 

        yum update --disablerepo=mongodb-org-3.0,dockerrepo,epel
    4. Enable Specific Repositories While Disabling Others:

      • Combine enabling and disabling repositories for fine-grained control: 

        yum update --disablerepo=* --enablerepo=mongodb-org-3.0
  • Notes:
    • Excluding repositories is useful when you want to avoid conflicts or unnecessary updates from specific repositories.
    • The `–disablerepo` option is temporary and only applies to the current command. For persistent changes, modify repository configuration files under `/etc/yum.repos.d/`.
  • References:
    • [The Geek Stuff: Managing Yum Repositories](https://www.thegeekstuff.com/2016/05/yum-update-disable-repo/)

10. Delete the File Content (Large) Without Opening It

  • Objective: Clear the content of a file without deleting or recreating the file itself. This is useful for managing large log files like `access.log`.
  • Methods:
    1. Redirect the File to Null:

      • This effectively empties the file: 

        > access.log
    2. Redirect `/dev/null` to the File:

      • Another way to clear the file's content: 

        cat /dev/null > access.log
    3. Copy `/dev/null` to the File:

      • This works similarly to the above: 

        cp /dev/null access.log
    4. Use `dd` Command:

      • A versatile method to achieve the same result: 

        dd if=/dev/null of=access.log
  • Notes:
    • These methods only empty the file but retain its metadata such as permissions and ownership.
    • Make sure the file is not being actively written by another process when clearing its content to avoid inconsistencies.
  • References:
    • [TecMint: Empty/Delete File Content Without Opening It](https://www.tecmint.com/empty-delete-file-content-linux/)

11. How to Deal with Stopped Jobs

  • Overview:
    • A stopped job is a process that has been temporarily moved to the background and paused, but it still occupies system resources (e.g., memory).
  • Common Commands:
    1. List Running and Stopped Jobs:

      • Displays all active and stopped jobs in the current shell session: 

        jobs
    2. Resume a Stopped Job in the Foreground:

      • Use the job ID to bring it back to the foreground: 

        fg 1
      • Replace `1` with the relevant job ID listed by the `jobs` command.
    3. Send a Job to the Background:
      • You can stop a job by pressing `Ctrl+Z`, which sends it to the background.
  • Notes:
    • If a job is no longer needed, you can terminate it using `kill` with the appropriate job ID.
    • Use `bg` to resume a stopped job in the background instead of bringing it to the foreground.
  • References:
    • [Unix StackExchange: Handling Stopped Jobs](https://unix.stackexchange.com/questions/116959/there-are-stopped-jobs-on-bash-exit)

12. What is Load Average in Linux

  • Overview:
    • Load average in Linux is a measure of CPU utilization at any given time.
    • It reflects the number of processes actively using or waiting for CPU time.
    • An idle system has a load of `0`. Each active or queued process adds `1` to the load.
  • Key Details:
    1. Load Average Values:
      • Load average is usually represented as three numbers (e.g., `0.03, 0.03, 0.01`):
        • First Value: Average CPU load over the last minute.
        • Second Value: Average CPU load over the last 5 minutes.
        • Third Value: Average CPU load over the last 15 minutes.
      • These averages help track short-term and long-term CPU trends.
    2. Understanding Load Average:
      • A load average of `1.0` on a single-core CPU means the CPU is fully utilized.
      • For multi-core CPUs, divide the load average by the number of cores to assess utilization.
        • Example: On a 4-core system, a load average of `4.0` indicates full utilization, while `2.0` suggests half utilization.
    3. Why Load Average is Useful:
      • It helps monitor system performance and identify bottlenecks.
      • Load averages change quickly as processes start and stop, making it a dynamic performance metric.
  • Example Output:

    • Command: 

      uptime

    • Output: 

      17:45:26 up 3 days, 4:12,  2 users,  load average: 0.03, 0.03, 0.01
      • Interpretation:
        • System time: `17:45:26`.
        • System uptime: `3 days, 4 hours, 12 minutes`.
        • Number of users logged in: `2`.
        • Load averages:
          • 1-minute: `0.03` (very low usage).
          • 5-minute: `0.03` (very low usage).
          • 15-minute: `0.01` (nearly idle).

    • Using `top`: 

      top

    • Sample `top` Output: 

      top - 17:46:38 up 3 days, 4:13,  2 users,  load average: 0.03, 0.03, 0.01
Tasks: 110 total,   1 running, 109 sleeping,   0 stopped,   0 zombie
%Cpu(s):  1.0 us,  0.5 sy,  0.0 ni, 98.5 id,  0.0 wa,  0.0 hi,  0.0 si,  0.0 st
KiB Mem :  8012340 total,  1223344 free,  3445784 used,  3343212 buff/cache
KiB Swap:  2097148 total,  2097148 free,        0 used.  4343212 avail Mem
      • Interpretation:
        • Load averages: `0.03`, `0.03`, `0.01` (low utilization).
        • CPU utilization: 1% user processes, 0.5% system processes, 98.5% idle.
        • Memory: Total `8GB`, used `3.4GB`, free `1.2GB`.
  • Tools to Monitor Load Average:

    • `uptime`: Displays the current time, system uptime, and load averages. 

      uptime

    • `top`: Provides real-time CPU, memory, and load average information. 

      top
    • `htop`: An enhanced version of `top` with an interactive interface.
  • References:
    • Linux Manual Pages for `uptime`, `top`, and `htop`.

13. How to Understand the Reason Why the Server Rebooted

  • Log File Locations:
    • For CentOS/RHEL systems, check the logs at `/var/log/messages`.
    • For Ubuntu/Debian systems, check the logs at `/var/log/syslog`.
    • Use the `tail` command or a text editor to filter or search for specific information about reboots.
  • Analyze Systemd Journal:

    • List all logged boots: Use `journalctl` to list the boots recorded by the system: 

      journalctl --list-boots
      • This command displays a list of previous boots with an index number (`-1`, `-2`, etc.).

      • Example output: 

        -2 4c3b5b2f2a4c401c8506ab65e03b3df7 Mon 2024-11-10 09:15:00 UTC—Mon 2024-11-10 17:35:00 UTC
-1 1a2f33ef3d3c401dbe36b5647e03fdf7 Mon 2024-11-11 09:20:00 UTC—Mon 2024-11-11 17:40:00 UTC
 0 c2a3b5b2f2a3c4dc6506ab56e03c3de5 Tue 2024-11-12 08:00:00 UTC—Tue 2024-11-12 16:20:00 UTC

    • Analyze a specific boot: Use the boot index to examine logs from that boot: 

      journalctl -b {num} -n
      • Replace `{num}` with the boot index (e.g., `-1` for the previous boot).
      • Use `-n` to limit the number of lines displayed.
  • Check When the Server Rebooted:

    • Use the `who` command to find the last reboot time: 

      who -b

      • Example output: 

        system boot  2024-11-12 08:00

    • Alternatively, use the `last` command to view reboot history: 

      last -x | head | tac

      • Example output: 

        reboot   system boot  5.15.0-43-generic Fri Nov 10 09:15 - 17:35  (08:20)
reboot   system boot  5.15.0-43-generic Mon Nov 12 08:00 - 16:20  (08:20)
  • Tips:

    • Use `grep` with `journalctl` to filter messages related to shutdown or restart: 

      journalctl | grep -i "shutdown"
journalctl | grep -i "reboot"
  • References:
    • `man journalctl`
    • Linux system logs documentation

14.1. Troubleshooting: Stale Output from `who` Command

  • Problem:
    • `who` command may not show current logged-in users but outdated data.
    • The file `~/var/run/utmp` might still fill up with login information.
  • Solution:

    • Look for defunct "login" processes and kill them: 

      ps -aux | grep defunct | awk '{print $2}' | xargs kill -9
  • Reference:

21. How to Partition, Mount, and Unmount Disks

  • Definitions:
    • MBR (Master Boot Record):
      • Traditional partition table with a 2TB limit and 4 primary partitions.
    • GPT (GUID Partition Table):
      • Modern table with support for volumes larger than 2TB and unlimited partitions.
    • Cluster: Minimum space a file occupies on disk.
    • Volume: A storage area with a single filesystem across one or multiple devices.
    • Partition: Logical division of a disk to create independent storage spaces.

  • Using `parted` for Disk Management: 

    # List all connected disks:
sudo parted -l 

# Connect to a specific disk (e.g., /dev/sdc):
sudo parted /dev/sdc

# Inside the parted environment:
mklabel gpt  # Set GPT as the partition table.

# Create a primary ext4 partition from 1 MiB to 8 GiB:
mkpart primary ext4 1MIB 8GIB

# Create a primary NTFS partition for the remaining space:
mkpart primary ntfs 8GIB 100%

quit  # Exit the parted environment.

# Format the created partitions:
sudo mkfs -t ext4 /dev/sdc1
sudo mkfs -t ntfs /dev/sdc2

# Mount the partitions:
sudo mount /dev/sdc1 /media/sbilmis/myext4
sudo mount /dev/sdc2 /media/sbilmis/myntfs

# Unmount the partitions:
sudo umount /media/sbilmis/myext4
sudo umount /dev/sdc1

# Enable a swap partition (if applicable):
sudo mkswap /dev/sdc3
sudo swapon /dev/sdc3

# View all storage devices in a tree-like format:
lsblk
  • Persistent Mounts:
    • Edit `/etc/fstab` to configure partitions for automatic mounting during boot.
  • Additional Notes:
    • Hard links in Linux reference file records, not file names. Renaming the original file does not break the link.

30.4. How to Connect/Disconnect from VPN via Command Line

  • Use the `nmcli` command to manage VPN connections on Network Manager-enabled systems.
  • Commands:

    • List available connections: 

      nmcli con

    • Activate a VPN connection by name: 

      nmcli con up id ConnectionName
  • Troubleshooting VPN Issues:
    • If you encounter `Error: Connection activation failed: no valid VPN secrets`:

      1. Edit the VPN configuration file: 

        sudo nano /etc/NetworkManager/system-connections/ConnectionName

      2. Under the `[vpn]` section, update the `password-flags` to: 

        password-flags=0

      3. Add the following block below `[vpn]`: 

        [vpn-secrets]
password=YourPassword

      4. If you changed `Xauth password-flags`, also add: 

        Xauth password=YourPassword

      5. Restart the Network Manager service: 

        sudo systemctl restart NetworkManager.service
  • Reference:
    • [AskUbuntu: VPN Connection via Command Line](https://askubuntu.com/questions/57339/connect-disconnect-from-vpn-from-the-command-line)
